01 System Architecture Overview

How every component connects end-to-end

Crossfire is a transparent security proxy that intercepts MCP (Model Context Protocol) and A2A (Agent-to-Agent) traffic between AI IDEs/SDKs and tool servers. It runs 20+ threat detectors on every message, streams events to a FastAPI hub, and a React dashboard visualizes everything in real-time.

02 MCP Stdio Proxy Flow MCP

proxy/proxy.py · proxy/protocol.py · proxy/proxy_stdio_main.py

The core man-in-the-middle: the IDE thinks it's talking to the MCP server directly, but crossfire-proxy sits in between, inspecting every JSON-RPC message on stdin/stdout.

sequenceDiagram participant IDE as IDE (Cursor/VS Code) participant Proxy as crossfire-proxy participant Det as Detectors + Policy participant Srv as MCP Server (child) participant Dash as Dashboard :9999 Note over IDE,Srv: Startup IDE->>Proxy: spawn crossfire-proxy -- npx server Proxy->>Srv: spawn child process (original cmd) Proxy->>Proxy: check_server_binding() Proxy->>Dash: resolve dashboard URL Note over IDE,Srv: Request Flow (IDE to Server) IDE->>Proxy: stdin JSON-RPC (tools/call) Proxy->>Proxy: protocol.read_message + classify Proxy->>Proxy: unicode_normalize arguments Proxy->>Det: Run 15+ request detectors Det-->>Proxy: list of Threats Proxy->>Proxy: Policy evaluate (block/allow) alt Guardian=BLOCK + critical/high threat Proxy-->>IDE: JSON-RPC error (blocked) Proxy->>Dash: POST /api/events (blocked=true) else Allow Proxy->>Srv: forward request via stdout pipe Proxy->>Dash: POST /api/events (threats attached) end Note over IDE,Srv: Response Flow (Server to IDE) Srv->>Proxy: stdout JSON-RPC response Proxy->>Det: Response detectors (sensitive_data, rug_pull) Det-->>Proxy: list of Threats Proxy->>IDE: forward response via stdout Proxy->>Dash: POST /api/events Dash-->>Proxy: guardian_mode in response
  1. IDE's MCP config is rewired to crossfire-proxy by the installer. The original server command is passed after --.
  2. proxy_stdio_main.py parses CLI args and calls asyncio.run(run_proxy(...)).
  3. proxy.py → run_proxy() spawns the real MCP server as a subprocess and creates two async tasks: ide_to_server (request path) and server_to_ide (response path).
  4. protocol.py reads newline-delimited JSON-RPC from stdin/stdout streams, classifies each message as request / notification / response.
  5. For tools/call requests: unicode normalization → 15+ detectors → policy check → optional Gemini enrichment → make_event() → broadcast() to dashboard.
  6. If Guardian mode is BLOCK and severity is critical/high, a JSON-RPC error is returned to the IDE instead of forwarding to the server.
  7. Responses from the server pass through response-side detectors (sensitive data leaks, rug-pull hash changes, resource poisoning).
  8. Every POST /api/events response from the dashboard carries the current guardian_mode, keeping the proxy synchronized.

03 MCP HTTP Proxy Flow MCP

proxy/mcp_http_proxy.py

For remote MCP servers that use HTTP/SSE transport instead of stdio. A FastAPI reverse proxy that applies the same detection pipeline.

sequenceDiagram participant Client as SDK / Agent participant Proxy as MCP HTTP Proxy participant Det as Detectors participant Up as Remote MCP Server participant Dash as Dashboard :9999 Client->>Proxy: POST /mcp (JSON-RPC body) Proxy->>Proxy: Parse JSON body Proxy->>Det: detect_request_threats() Det-->>Proxy: list of Threats alt Blocked by Guardian Proxy-->>Client: JSON-RPC error Proxy->>Dash: POST /api/events (blocked) else Allowed Proxy->>Up: httpx POST (forward body) Up-->>Proxy: JSON response OR SSE stream alt SSE stream loop Each SSE data line Proxy->>Det: detect_response_threats() Proxy-->>Client: stream SSE line end else JSON response Proxy->>Det: detect_response_threats() Proxy-->>Client: return JSON end Proxy->>Dash: POST /api/events end
  1. SDK or agent sends HTTP POST to the local proxy endpoint instead of the remote MCP server.
  2. Proxy parses the JSON-RPC body and runs request-side detectors (same set as stdio minus cross_call tracking).
  3. Optionally queries Gemini for AI-powered analysis on suspicious patterns.
  4. Forwards to the upstream MCP server via httpx.
  5. For SSE responses, each data: line is parsed and scanned for response-side threats.
  6. Can hide blocked tools from tools/list responses when configured.

04 A2A Protocol Proxy Flow A2A

proxy/a2a_proxy.py

Intercepts Agent-to-Agent protocol traffic (Google's agent interoperability standard). Monitors agent cards, message parts, and session integrity.

sequenceDiagram participant A1 as Client Agent participant Proxy as A2A Proxy :9998 participant Det as A2A Detectors participant A2 as Upstream Agent participant Dash as Dashboard Note over A1,A2: Agent Card Discovery A1->>Proxy: GET /.well-known/agent.json Proxy->>A2: Forward GET A2-->>Proxy: agent.json Proxy->>Det: detect_a2a_impersonation (hash card) Proxy-->>A1: Return card Note over A1,A2: message/send A1->>Proxy: POST / method=message/send Proxy->>Det: A2A injection + exfiltration scan Proxy->>A2: Forward JSON-RPC A2-->>Proxy: Response with artifacts Proxy->>Det: Scan response + session_smuggling Proxy-->>A1: Return response Proxy->>Dash: POST /api/events Note over A1,A2: message/stream SSE A1->>Proxy: POST / method=message/stream Proxy->>A2: Forward + stream SSE loop Each SSE event A2-->>Proxy: data: status/artifact Proxy->>Det: Scan parts + session tracking Proxy-->>A1: Forward SSE event end Proxy->>Dash: POST /api/events
  1. A2A proxy listens on port 9998 (configurable in crossfire.yaml).
  2. Agent card requests are forwarded and hashed; subsequent card fetches are compared for impersonation (hash change detection).
  3. Message parts (text, data, file) are scanned for injection and exfiltration patterns.
  4. SessionSmuggleDetector tracks per-session message ratios and timing to detect server-side message injection.
  5. SSE streams (message/stream) are intercepted line-by-line for real-time detection.

05 Threat Detection Pipeline SECURITY

proxy/detectors/ — 20+ modules

Every message through Crossfire passes through a multi-stage detection pipeline. Detectors run on both request and response paths.

  1. Unicode normalization — converts homoglyphs and NFC so detectors see clean ASCII-like text.
  2. Request detectors — run on every tools/call: shell injection patterns, sensitive file paths, credential tokens, SQL injection, path traversal, cross-call chains, config poisoning, session fixation, cross-tenant violations, OAuth abuse, neighbor hijacking.
  3. Response detectors — run on tools/list and tools/call responses: leaked secrets, tool description injection, schema poisoning, rug-pull (tool hash changes), resource poisoning, typosquatting.
  4. Gemini enrichment (optional) — if GOOGLE_API_KEY is set, a rolling context buffer is sent to Gemini for secondary AI analysis. Adds gemini_analysis with confidence + explanation to threats.
  5. Policy engine — evaluates per-server/per-tool rules + severity thresholds. Can block before forwarding.
  6. Event builder — normalizes everything into a CrossfireEvent dict and POSTs to the dashboard.

06 Guardian Mode Toggle CONTROL

server/guardian.py · proxy/proxy.py · dashboard App.tsx

Guardian mode switches between MONITOR (observe only) and BLOCK (actively prevent critical/high threats). The state is synchronized across all three layers.

sequenceDiagram participant UI as React Dashboard participant WS as WebSocket /ws participant API as FastAPI Server participant G as Guardian State participant Proxy as MCP Proxy Note over UI,Proxy: User toggles Guardian UI->>WS: set_guardian command, mode=block WS->>API: websocket_endpoint handler API->>G: guardian.set_mode(block) G-->>API: mode = block API->>WS: broadcast guardian_update WS-->>UI: Update UI to BLOCK state Note over UI,Proxy: Proxy syncs via event POST Proxy->>API: POST /api/events API-->>Proxy: guardian_mode=block in response Proxy->>Proxy: Update _guardian_mode Note over UI,Proxy: Proxy enforces block Proxy->>Proxy: Threat detected severity=critical Proxy->>Proxy: guardian_mode == block so DENY Proxy-->>UI: JSON-RPC error returned to IDE
  1. User clicks the Guardian toggle in the dashboard header.
  2. Dashboard sends {command: "set_guardian", mode: "block"} over WebSocket.
  3. Server updates the Guardian singleton and broadcasts guardian_update to all connected WS clients.
  4. The proxy learns the new mode through the guardian_mode field in POST /api/events responses.
  5. In BLOCK mode, the proxy returns JSON-RPC errors for critical/high-severity threats instead of forwarding to the MCP server.
  6. REST API also available: GET /api/guardian (read) and POST /api/guardian (set).

07 Dashboard Server (FastAPI) SERVER

server/main.py · server/events.py · server/guardian.py

The hub that connects proxies to the UI. Ingests events via REST, stores them in a bounded deque, broadcasts to WebSocket clients, and serves the built React app.

  1. Rate limiter — per-IP sliding window caps ingestion at ~200 events/sec to prevent flooding.
  2. EventStore — bounded deque(maxlen=10000) with appendleft (newest first). Per-server stats track calls, threats, tools discovered, sources, and last seen time.
  3. ConnectionManager — maintains a list of active WebSocket connections. broadcast() sends JSON to all; silently drops dead sockets.
  4. Every POST /api/events validates EventIn → store.add() → manager.broadcast() → responds with current guardian_mode.
  5. WebSocket clients can send set_guardian commands; server broadcasts guardian_update to all.
  6. Scan endpoint — POST /api/scan runs the scanner in the background; progress is pushed over WebSocket.
  7. Static serving — if a built dashboard exists at server/web_dist/ or dashboard/dist/, it is mounted at /; otherwise a stub HTML page is served.

10 Scanner Workflow

proxy/scanner.py

Active MCP vulnerability scanner — spawns servers, probes tools, and generates security reports without going through the live proxy.

sequenceDiagram participant CLI as crossfire scan / API participant Scan as proxy/scanner.py participant Srv as MCP Server subprocess participant Det as Detectors CLI->>Scan: scan_server(command, name) Scan->>Srv: spawn subprocess Scan->>Srv: initialize request Srv-->>Scan: initialize response Scan->>Srv: tools/list request Srv-->>Scan: tools/list response Scan->>Det: tool_scanner (description poisoning) Scan->>Det: schema_poisoning (inputSchema) loop For each tool Scan->>Scan: _build_probe_arguments Scan->>Srv: tools/call synthetic probe Srv-->>Scan: response Scan->>Det: rules + sensitive_data + cross_call end Scan->>Scan: Aggregate ScanFindings Scan-->>CLI: ScanReport
  1. Scanner spawns the MCP server as a child process using the resolved command.
  2. Sends initialize to establish the MCP session.
  3. Sends tools/list and runs description poisoning + schema poisoning detectors on each tool definition.
  4. For each tool, builds synthetic probe arguments from the JSON Schema and sends tools/call.
  5. Runs detectors on both the probe requests and server responses.
  6. Aggregates all findings into a ScanReport with severity counts and per-tool results.
  7. Can be triggered from CLI (crossfire scan) or via the dashboard API (POST /api/scan) with WebSocket progress updates.

12 npm / Python Bridge

bin/ · package.json · pyproject.toml

Crossfire is published on both npm and PyPI. The npm package is a thin JavaScript wrapper that resolves and spawns the Python CLI.

  2. cli.js finds Python 3.10+, avoids self-recursion, and spawns the real crossfire Python CLI.

  4. PyPI console scripts — crossfire maps to proxy.__main__:main, crossfire-proxy maps to proxy.proxy_stdio_main:main.
  5. The dual-publish ensures users can install via npm install crossfire-mcp or pip install crossfire-mcp — both paths lead to the same Python core.
